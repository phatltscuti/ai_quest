Google Workspace CLI (gws): Tự động hóa Drive/Sheets bằng lệnh terminal

Ngày viết: 20/03/2026 • Nguồn: googleworkspace/cli • YouTube reference video

Mục tiêu bài viết: (1) Tóm tắt core features của CLI dựa trên repo chính thức, (2) làm 2 ví dụ “chạy thật” trong máy và ghi lại command + output, (3) đưa ra góc nhìn cá nhân về cách áp dụng để tối ưu công việc hằng ngày.

1. Core features có thể đạt được với Google Workspace CLI (`gws`)

Theo README chính thức của dự án, CLI mang tên gws đóng vai trò như “một CLI cho mọi Google Workspace”: thao tác được nhiều dịch vụ (Drive, Gmail, Calendar, Sheets, Chat, ...), dùng chung một cú pháp lệnh và luôn trả về output dạng JSON có cấu trúc.

1.1. Tự động sinh command từ Discovery Service

Điểm khác biệt lớn: gws “không ship một danh sách lệnh tĩnh”. Nó đọc Discovery Service của Google tại runtime và tự dựng toàn bộ bề mặt lệnh. Vì vậy khi Google Workspace thêm API/method mới, gws có thể tự cập nhật command surface mà không cần bạn “chờ release CLI”.

1.2. Output JSON + khả năng script hóa

README nhấn mạnh mọi output (kết quả và error) đều có cấu trúc JSON. Điều này giúp bạn dễ dàng đưa vào pipeline (ví dụ: pipe sang công cụ khác, hoặc feed cho AI agent).

1.3. `--dry-run` để validate trước khi gọi API

Nhiều lệnh hỗ trợ --dry-run để “validate request cục bộ” (không gửi đi). Trong thực tế, nó cực hữu ích để kiểm tra tham số URL/query/body sẽ trông như thế nào trước khi bạn đăng nhập/đã set credentials.

1.4. Helper commands (prefix `+`) cho workflow thực dụng

Bên cạnh Discovery-generated commands, repo có các “helper commands” tiền tố + để đưa ra thao tác “đúng nhu cầu người dùng”: ví dụ gmail +send, sheets +append, calendar +agenda, drive +upload, ... Những lệnh này thường dùng tham số dạng cờ/flag đơn giản hơn so với JSON payload.

1.5. AI Agent Skills đi kèm

Repo mô tả có 40+ (và thực tế có thể lên đến 100+ theo nội dung video) “agent skills” và có cả index/skills để cài đặt. Ý nghĩa thực tế: bạn có thể để LLM/AI agent gọi trực tiếp các thao tác Workspace mà không tự viết “glue code” cho từng REST endpoint.

1.6. Authentication đa luồng + env vars

README mô tả nhiều workflow: gws auth setup, gws auth login, hoặc truyền credentials qua GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE, token qua GOOGLE_WORKSPACE_CLI_TOKEN, hoặc service account.

1.7. Introspect schema & lộ “hợp đồng API”

Bạn có thể dùng gws schema <service>.<resource> để xem mô tả method, request/response schema. Đây là cách nhanh để hiểu “phải truyền body/query thế nào” trước khi bạn ghép automation.

2. Thử nghiệm thực tế trong máy: command + execution result

2.1. Cài đặt và kiểm tra version

npm install -g @googleworkspace/cli
gws --version
gws 0.18.1
This is not an officially supported Google product.

2.2. Ví dụ #1 (Drive): Dry-run “list files”

Mục tiêu: kiểm tra CLI dựng đúng URL/query request khi gọi Drive API. Mình dùng --dry-run để tránh phụ thuộc credentials. 

gws drive files list --params '{}' --dry-run
{
  "body": null,
  "dry_run": true,
  "is_multipart_upload": false,
  "method": "GET",
  "query_params": [],
  "url": "https://www.googleapis.com/drive/v3/files"
}
Nhận xét nhanh: Result cho thấy gws dựng request rõ ràng (method, url, query_params). Điều này đủ để bạn “soi” logic before/after khi thay tham số trong automation.

2.3. Ví dụ #2 (Sheets): Helper +append thêm 1 dòng (dry-run)

Mục tiêu: dùng helper command để append dữ liệu vào Sheets mà không cần truyền JSON phức tạp. Mình chạy với --dry-run và dùng một ID giả để chỉ kiểm tra phần request. 

gws sheets +append --spreadsheet TEST_SPREADSHEET_ID --values 'Alice,100' --dry-run
{
  "body": {
    "values": [
      [
        "Alice",
        "100"
      ]
    ]
  },
  "dry_run": true,
  "is_multipart_upload": false,
  "method": "POST",
  "query_params": [
    [
      "valueInputOption",
      "USER_ENTERED"
    ]
  ],
  "url": "https://sheets.googleapis.com/v4/spreadsheets/TEST%5FSPREADSHEET%5FID/values/A1:append"
}

2.4. Ví dụ #3 (Schema): xem “contract” của thao tác tạo Spreadsheet

Trong video reference có nhắc đến việc gom dữ liệu từ Drive rồi đưa vào Sheets để tracking. Thay vì đoán cấu trúc request để tạo spreadsheet, bạn có thể dùng gws schema ... để xem schema cần truyền. 

gws schema sheets.spreadsheets.create
{
  "description": "Creates a spreadsheet, returning the newly created spreadsheet.",
  "httpMethod": "POST",
  "parameters": {},
  "path": "v4/spreadsheets",
  "requestBody": {
    "schema": {
      "description": "Resource that represents a spreadsheet.",
      "properties": {
        "dataSourceSchedules": {
          "description": "Output only. A list of data source refresh schedules.",
          "items": {
            "$ref": "DataSourceRefreshSchedule"
          },
          "readOnly": true,
          "type": "array"
        },
        "dataSources": {
          "description": "A list of external data sources connected with the spreadsheet.",
          "items": {
            "$ref": "DataSource"
          },
          "type": "array"
        },
        "developerMetadata": {
          "description": "The developer metadata associated with a spreadsheet.",
          "items": {
            "$ref": "DeveloperMetadata"
          },
          "type": "array"
        },
        "namedRanges": {
          "description": "The named ranges defined in a spreadsheet.",
          "items": {
            "$ref": "NamedRange"
          }
      ... (đã rút gọn)

2.5. Ghi chú về một giới hạn trong môi trường PowerShell

Khi mình thử truyền payload JSON có key (ví dụ --params '{"pageSize": 2}'), CLI báo lỗi validation kiểu “key must be a string…”, và khi debug qua argv mình thấy dấu " trong JSON bị “nuốt” khi truyền vào native app. Vì vậy, để đảm bảo demo chạy được trong môi trường hiện tại, mình ưu tiên: --params '{}' (không có key), hoặc dùng helper commands (+append) vốn nhận tham số dạng chuỗi/flag.

3. Quan điểm cá nhân: dùng `gws` để tối ưu công việc hằng ngày

Với workflow thường ngày của mình (kiểu: “mỗi sáng/tuần cần gom dữ liệu từ Drive rồi cập nhật Sheets để theo dõi”), mình thấy gws có thể đóng vai trò như một “tầng tự động hóa” ở mức thấp nhưng rất linh hoạt:

  • Giảm thao tác lặp: dùng lệnh kiểu drive files list --dry-run để chuẩn hóa truy vấn, rồi đến lúc thật mình chạy không dry-run để pull danh sách file cần xử lý.
  • Ghi log/aggregation nhanh: helper sheets +append phù hợp để lưu kết quả tổng hợp (ví dụ: tên file mới, link, trạng thái, người phụ trách) vào 1 bảng tracking.
  • AI agent làm “thư ký”: vì output dạng JSON có cấu trúc, agent có thể đọc/ra quyết định dựa trên schema thay vì phải parse text thủ công. Điều này đặc biệt hữu ích khi bạn muốn “hỏi AI” và để AI tự chọn action (ví dụ: tìm tài liệu liên quan trong Drive rồi ghi vào Sheet).
Best practice mình sẽ áp dụng: luôn bắt đầu bằng --dry-run và gws schema ... trước, sau đó mới chuyển sang automation thật. Cách này giúp giảm rủi ro “đụng nhầm API/method” và làm quá trình thiết kế workflow nhanh hơn.
